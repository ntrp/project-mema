Mema includes a DLNA Digital Media Server surface for televisions and UPnP
clients. The current implementation is protocol-first and intentionally shares
media delivery primitives with browser preview and VLC playback where possible.

The long-term direction is tracked by the DLNA profile epic and PRDs:

- `docs/prd/dlna-upnp-clean-room-spec.md`
- `docs/prd/dlna-device-profiles-clean-room-spec.md`
- `docs/prd/dlna-ums-feature-parity-plan.md`
- GitHub epic `#109`

## Runtime Shape

The DLNA manager owns discovery, status, diagnostics, profile selection, and
resource delivery coordination.

```txt
SettingsStore
  -> internal/dlna.Manager
      -> SSDP advertiser/responder
      -> HTTP device and service documents
      -> SOAP dispatcher
      -> ContentDirectory tree
      -> ConnectionManager
      -> artwork, subtitle, and media resources
      -> renderer profile matching
      -> stream/transcode diagnostics
```

The app HTTP router mounts DLNA paths before the frontend fallback so televisions
can request root-level UPnP paths without receiving the Svelte application.
Handlers are exposed both at root paths and under `/dlna`:

```txt
/rootDesc.xml
/contentDirectory.xml
/connectionManager.xml
/mediaReceiverRegistrar.xml
/control/content-directory
/control/connection-manager
/control/media-receiver-registrar
/resource/{objectID}
/artwork/{objectID}
/subtitle/{objectID}
/events/{service}

/dlna/rootDesc.xml
/dlna/contentDirectory.xml
/dlna/resource/{objectID}
```

## Discovery

SSDP is configured from DLNA settings:

- enabled state
- friendly name
- selected network interfaces
- announce interval
- stable device UUID
- allowed CIDRs

When enabled, the SSDP runtime advertises interface-specific descriptor URLs.
The URL must be reachable by the renderer, so it is based on the selected LAN
interface rather than `0.0.0.0`.

The status endpoint reports:

- running state
- bound interfaces
- advertised URLs
- available interfaces
- last SSDP event
- last error
- recent clients
- active streams and transcodes

## Device Documents

The root descriptor describes Mema as a UPnP MediaServer and points clients to
service descriptors. It includes compatibility details that real clients expect,
including icons and the optional MediaReceiverRegistrar service.

Service descriptor documents are generated by Mema and served with explicit XML
content type, content length, and connection-close behavior for renderer
compatibility.

## SOAP Services

SOAP requests are dispatched by service URN and action name.

Implemented service areas:

- ContentDirectory
  - `GetSearchCapabilities`
  - `GetSortCapabilities`
  - `GetSystemUpdateID`
  - `Browse`
  - `Search`
- ConnectionManager
  - source protocol information filtered by renderer profile
- MediaReceiverRegistrar
  - compatibility service for clients that expect it

SOAP handlers render DIDL-Lite from the content tree and resource planner. They
also record diagnostics such as last SOAP action, client profile, object ID, and
errors.

## Content Tree

The ContentDirectory tree maps Mema library items to stable opaque object IDs.
Filesystem paths are never exposed as object IDs.

Root containers currently include:

- Movies
- TV Shows
- Collections
- Recently Added
- Recently Updated
- Genres
- Years

Only containers with visible media are shown at the root. Movies with one file
can be exposed directly as video items; multi-file media and series can expose
container children.

DIDL-Lite output includes:

- standard DIDL, UPnP, Dublin Core, and compatibility namespaces
- containers and video items
- titles, classes, dates, genres, artists, albums
- artwork URLs
- media resources
- subtitle resources
- resource attributes from probe metadata where available

## Resource Delivery

The resource endpoint resolves an object ID back to a visible media file and
serves one of these delivery modes:

| Mode | Current behavior |
| --- | --- |
| Direct file | Uses shared range-capable file serving. |
| HLS playlist | Uses shared delivery segment planning for HLS-capable profiles. |
| HLS segment | Streams an ffmpeg-generated MPEG-TS segment. |
| LG audio transcode | Copies compatible video and transcodes unsupported audio to AAC in Matroska. |
| Cached remux | Builds or reuses a seekable cached Matroska output for real byte-range seeks. |

Initial play is treated differently from real seek. `Range: bytes=0-` is an
initial play request and should stream immediately. Nonzero byte ranges are
treated as seek requests and can use a cached remux/transcode output.

Transcode work is bounded by slots, uses safe media-tool path validation, and
records active stream/transcode diagnostics.

## Renderer Profiles Today

Renderer profiles are currently Go structs in `internal/dlna/profile.go`.

Current seeded profiles:

- Generic DLNA
- VLC
- Kodi
- Samsung TV
- LG TV
- Sony TV
- BubbleUPnP
- Chromecast

Matching currently uses:

1. manual IP override
2. User-Agent, friendly name, and HTTP header token matching
3. sticky IP reuse from recent clients
4. generic fallback

Sticky IP reuse is important because many televisions send useful DLNA headers
for SOAP but weaker browser-like headers for media requests. Once a client IP
is identified as LG, later resource requests from the same IP can keep the LG
profile.

Profiles currently affect:

- HLS preference or avoidance
- eventing disablement
- subtitle format list
- response headers
- advertised source protocol order
- browser/WebKit delivery profile selection
- LG audio-transcode compatibility behavior

## Planned Profile Architecture

Epic `#109` replaces hardcoded renderer profiles with persisted, seeded, and
editable renderer profiles.

Future profile matching order:

1. manual device override by UUID
2. manual device override by IP
3. sticky connected-renderer association by IP
4. UPnP UUID and device details
5. HTTP headers
6. default renderer profile

Future profile data is structured rather than copied from UMS configuration
files. Each profile can define:

- identity and seed version
- vendor and device class
- match rules and priority
- direct-play capability rules
- delivery preferences
- DLNA protocol flags
- subtitle behavior
- artwork and image behavior
- metadata and tree compatibility flags
- compatibility notes and confidence level

Seeded profiles must cover the UMS-supported device families in clean-room
form. User edits must survive seed upgrades, and each seeded profile must be
resettable or cloneable.

The storage layer now keeps two profile copies: seeded defaults and editable
runtime profiles. `internal/storage/seeds/defaults.sql` refreshes defaults on
startup and only refreshes runtime profiles that have not been customized.
Device overrides are stored separately and can bind a renderer profile by IP
address or renderer UUID.

The DLNA manager loads those rows into an in-memory matcher cache. Manual UUID
and IP overrides are checked before automatic matching. Automatic matching uses
the structured seed rules, then falls back to the recent-client sticky IP path
for weak media requests and finally the generic profile.

The selected profile also drives a capability evaluator. Probe metadata is
normalized into container, video codec, audio codec, and resolution checks.
The evaluator returns a delivery decision, reason codes, and trace rows that
feed DIDL resources, `ConnectionManager` protocol info, and diagnostics.

Settings API endpoints expose renderer profiles, profile clone/reset/import/
export actions, IP/UUID device overrides, and recent renderer devices. Mutating
profile or override endpoints refresh the DLNA manager cache so running
matching behavior follows saved settings.

## Settings UI

Settings > DLNA includes a Device Profiles panel over the profile storage and
matching APIs.

The panel contains:

- searchable seeded profile table
- recent devices table
- profile override selector by IP or UUID
- profile editor for matching, direct-play, delivery, DLNA flags, subtitles,
  artwork, metadata, and quirks
- clone, reset, enable/disable, import, and export actions
- decision trace viewer for a selected device and media file

Seeded profiles can be edited in place; reset restores the seed copy, while
clone creates a user-owned profile. Recent devices can be pinned to a selected
profile by IP, and manual overrides can target either IP or renderer UUID.
The trace viewer surfaces the effective profile, override, matching profile,
and profile rule payloads used when debugging renderer compatibility.

## Future Delivery Model

The profile engine will feed a structured capability evaluator. The evaluator
will compare probe metadata against profile rules and choose:

- direct play
- remux
- audio-only transcode
- full video transcode
- HLS

Resource advertisement must come from the same decision. A renderer should
never see a resource that Mema cannot serve.

The future delivery service should generalize the current LG-specific
Matroska audio transcode path into profile-specific targets. Seek mode,
container choice, audio codec, video codec, MIME type, DLNA flags, and subtitle
resource behavior should all come from the selected profile.

## Operational Rules

DLNA remains LAN-scoped:

- disabled by default
- interface allowlist
- CIDR allowlist
- optional per-device allow/deny in the future
- no absolute path exposure
- safe ffmpeg argument construction
- bounded stream and transcode counts

Diagnostics are part of the architecture, not a later UI nicety. Profile
matching and delivery decisions need trace output because renderer failures are
usually compatibility mismatches rather than simple server errors.
